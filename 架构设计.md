# NXCopilot — 软件架构设计与开发计划

> 基于 NXOpen Python API 开发 Windows 客户端，与 Siemens NX 同进程运行，
> 实现 AI Agent 驱动的正向设计、逆向设计、正逆向混合设计。

---

## 一、核心设计理念

### NXOpen Python 作为中间表示 — 用于分析，不用于重建

NXCopilot 的核心技术路线：

```
          +----------------------+
          |   NX 模型 (.prt)      |
          +----------+-----------+
                     | NXOpen API 读取
                     v
          +----------------------+
          |   特征树 (结构化)      |  <- NXOpen: Part.Features + Part.Expressions
          +----------+-----------+
                     | 转换为 LLM 可读的 NXOpen Python 代码
                     v
          +----------------------+
          | NXOpen 代码 (仅供分析) |  <- LLM 通过阅读代码理解模型设计
          +----------+-----------+
                     | LLM 分析需求，定位要修改的特征
                     v
          +----------------------+
          |  局部修改指令          |  <- 只修改目标特征，不重建全模型
          |  (改 Expression / 插入 / 删除) |     保持所有已有特征不变
          +----------+-----------+
                     | NXOpen 执行局部修改
                     v
          +----------------------+
          |  更新后的 NX 模型      |  <- 特征名称、引用、关联全部保留
          +----------------------+
```

**关键原则：**

| 原则 | 说明 |
|------|------|
| 特征树 -> NXOpen 代码用于分析，不用于执行 | LLM 读代码是为了理解模型结构，不是用来重建模型 |
| 修改永远局部 | 不改的特征不动，只针对目标特征生成修改代码 |
| Expression 优先 | NX 的 Expression 系统是参数化核心，优先通过它修改参数 |
| NXOpen 同进程执行 | Agent 与 NX 同进程运行，无 COM 桥接开销 |

### 与 CatiaCopilot 的技术路线对比

| 维度 | CatiaCopilot | NXCopilot |
|------|-------------|-----------|
| CAD 软件 | CATIA V5 | Siemens NX |
| 连接方式 | pywin32 COM (GetActiveObject) | NXOpen.Session.GetSession() (同进程) |
| 中间表示 | VBS 代码 | NXOpen Python 代码 |
| 参数修改 | FindObjectByName + 属性赋值 | Expression EditWithUnits |
| 特征创建 | catia-mcp-server (87 MCP 工具) | NXOpen Builder API (直接调用，无需 MCP) |
| 代码验证 | VBS 语法检查 (基础) | Python ast 模块 (完整语法树) |
| 执行方式 | COM ExecuteScript | exec() 同进程执行 |
| 特征类型覆盖 | 23 种 | 29 种 (持续扩展) |
| 单元测试 | 302 个 | 45 个 (离线模式) |

NXCopilot 的核心优势：NX 原生支持 Python (NXOpen)，不需要 COM 桥接层，
Agent 可以直接在 NX Journal 环境中运行，调用延迟从毫秒级 COM 降低到微秒级函数调用。

### 架构层次

| 层 | 技术 | 职责 |
|----|------|------|
| UI 层 | PySide6 (独立窗口，HTTP 连接 Agent) | 对话界面、特征树可视化、LLM 设置 |
| Agent 层 | Python/FastAPI (同机运行) | LLM 调用、意图路由、任务编排 |
| NX Engine 层 | 本地 Python 引擎 (无 LLM) | 读特征树 -> 转 NXOpen 代码 -> 生成局部修改指令 |
| NX Bridge 层 | NXOpen Session 管理 + 代码执行器 | 连接管理、执行重试、错误分类 |

与 CatiaCopilot 的层次差异：NXCopilot **没有工具层 (MCP Server)**。
NX 不需要 MCP 中间层 — NXOpen Python API 是一等公民，Agent 直接调用即可。
这简化了架构，减少了一层进程间通信。

## 三、三大核心能力

### 正向设计 — 自然语言驱动建模

```
用户: "在XY平面画一个50x30矩形，拉伸20mm，倒R3"
                           |
               +-----------+-----------+
               | LLM 解析需求           |
               | -> 生成 NXOpen 代码    |
               | -> NXOpen 执行创建     |
               | -> NX 中出现新特征      |
               +-----------------------+
```

正向设计通过 ForwardBuilder 生成 NXOpen Python 创建代码。
12 个 Builder 模板覆盖: Extrude, Revolve, Hole, EdgeBlend, Chamfer, Shell,
CircularPattern, RectangularPattern, Mirror, BooleanUnite, BooleanSubtract, DatumPlane。

### 意图路由 — 三种输入模式

Orchestrator 通过 `_classify_intent()` 将用户输入自动路由到三条路径：

| 意图 | 触发条件 | 处理方式 | 需要 NX |
|------|----------|----------|---------|
| general | 日常对话、知识咨询 | 直接调 LLM 聊天 | 否 |
| forward | 含"创建/画/拉伸/打孔"等关键词 | LLM 生成 NXOpen 创建代码 | 否 (离线可用) |
| nx_op | 含"修改/删除/参数/特征树"等关键词 | 读取模型 -> LLM 分析 -> NXOpen 执行 | 是 (无则降级为对话) |

NX 未连接时，general 和 forward 路径完全可用，nx_op 自动降级为对话模式。

### 逆向设计 — AI 读懂当前模型

```
用户: "这个法兰盘什么结构？"
                           |
               +-----------+-----------+
               | 1. NXOpen 读特征树     |
               | 2. 转 NXOpen 代码表示  |
               | 3. LLM 阅读代码分析    |
               | 4. LLM 理解设计意图    |
               | 5. 回答用户            |
               +-----------------------+
```

逆向不等于反向生成代码。逆向 = AI 能理解当前 NX 文件的设计状态。
NX Engine 把特征树转换为 NXOpen Python 代码（仅用于 LLM 阅读分析），
LLM 看代码就能理解模型的结构和参数 — "这是一个法兰盘，有基体、中心孔、螺栓孔阵列"。

后续对话时，用户说"在螺栓孔之间加加强筋"，AI 知道螺栓孔在哪、尺寸多少，
因为已经读过模型了。

### 正逆向混合设计 — 核心工作流

```
时间线:

1. 已有模型 (或新建)
   -> Agent 读取特征树，转为 NXOpen 代码供 LLM 分析 (不执行)

2. 用户："把螺栓孔从6个改成8个，直径M8改M10"
   -> LLM 阅读代码 -> 理解模型 -> 定位 CircularPattern 和 Hole
   -> 生成局部修改指令 (不改的特征不动):
        # 修改 CircularPattern 数量
        expr = work_part.Expressions.FindObject("pattern_count")
        unit = work_part.UnitCollection.FindObject("Number")
        work_part.Expressions.EditWithUnits(expr, unit, "8")

        # 修改 Hole 直径
        expr = work_part.Expressions.FindObject("hole_dia")
        unit_mm = work_part.UnitCollection.FindObject("MilliMeter")
        work_part.Expressions.EditWithUnits(expr, unit_mm, "10")
        work_part.Update()
   -> NXOpen 执行 -> NX 模型更新 (只有螺栓孔变了)

3. 用户在 NX 中手工调整了一个齿根圆角

4. 用户："再检查干涉情况"
   -> Agent 重新读特征树 -> 发现"圆角变了" (DeltaDetector)
   -> 更新分析结果 -> 回复用户

   用户："在干涉位置加大根部到 R8"
   -> LLM 定位到对应 EdgeBlend -> 生成 Expression 修改
   -> NXOpen 执行 -> 模型更新 -> 答复
```

混合设计的核心价值：在"自然语言修改"和"手工直接改"之间无缝切换，
通过每次对话前重新读取特征树来保证 Agent 的认知与 NX 实际状态一致。

## 四、NX Engine 模块详细设计

NX Engine 是 NXCopilot 的技术核心，所有不依赖 LLM 的逻辑都在这里。
与 CatiaCopilot 的 VBS Engine 对应，但利用 NX 原生 Python API 的优势做了简化。

### reader.py — NXOpen 特征树读取 (968 行)

通过 NXOpen API 遍历当前 Work Part 的全部特征、表达式、Body，输出 FeatureTreeSnapshot。

```
PartReader.read_feature_tree()
  -> NXOpen.Session.GetSession() 获取 Session
  -> session.Parts.Work 获取 Work Part
  -> Part.Features 遍历全部特征
  -> Part.Expressions 读取全局表达式
  -> Part.Bodies 读取 Body 列表
  -> 对每个特征: 类型 / 名称 / 参数 / 依赖关系 / 抑制状态
  -> 输出 FeatureTreeSnapshot
```

**与 CATIA reader 的关键差异：**
- CATIA: pywin32 COM 遍历，需要处理 COM 对象生命周期、类型转换
- NX: NXOpen 原生 Python 对象，直接属性访问，无桥接开销

数据模型（与 CatiaCopilot 通用）：

```
FeatureNode
  id: str             # 唯一标识 (NX: Tag)
  name: str           # 显示名称 (如 "EXTRUDE(1)")
  feature_type: str   # 特征类型 (Extrude / Hole / EdgeBlend / ...)
  parameters: dict    # 参数名 -> 值
  expressions: dict   # 关联的 NX Expression
  depends_on: list    # 依赖的其他特征名
  sketch_ref: str     # 关联的草图名称
  suppressed: bool    # 是否被抑制

FeatureTreeSnapshot
  part_name: str
  file_path: str
  features: list[FeatureNode]
  feature_count: int
  body_names: list[str]
  expressions: dict[str, float]   # 全局 Expression (NX 参数化核心)
  sketch_names: list[str]
  nx_version: str

FeatureTreeDelta
  added: list[FeatureNode]     # 新增特征
  modified: list               # 修改的特征 (参数变化)
  removed: list[str]           # 删除的特征名称
  unchanged: list[str]         # 不变的特征
```

**FeatureType 枚举覆盖 44 种 NX 特征类型：**
实体 (Extrude/Revolve/Sweep/Loft/Boolean)、孔槽 (Hole/Pocket/Slot/Groove)、
圆角倒角 (EdgeBlend/Chamfer/FaceBlend)、壳筋 (Shell/Rib/Boss)、
阵列 (CircularPattern/RectangularPattern/Mirror)、基准 (DatumPlane/DatumAxis)、
草图 (Sketch) 等。

### expression.py — NX Expression 管理器 (120 行)

**NX 独有模块，CATIA 无对应。**

NX 的 Expression 系统是参数化建模的核心，比 CATIA 的 Parameter 系统更统一：

```
# NX Expression 操作（参数化建模的核心接口）
expr = work_part.Expressions.FindObject("p1")        # 查找
work_part.Expressions.EditWithUnits(expr, unit, "25") # 修改
work_part.Expressions.CreateWithUnits("new_p", unit, "10") # 创建
```

ExpressionManager 封装了 read_all / find / edit / create / to_dict 五种操作，
是 editor.py 修改参数的首选路径。

### converter.py — 特征树 -> NXOpen 代码 (223 行)

把特征树转换为 LLM 可读的 NXOpen Python 代码。这份代码仅供 LLM 阅读理解模型结构，
绝对不用于重建 NX 模型。

```
feature_tree_to_nxopen(snapshot)
  -> 输出带注释的特征列表（类型/参数/依赖/表达式）
  -> 包含全局 Expression 列表
  -> 包含 Body 列表
  -> 输出: 可供 LLM 阅读分析的代码字符串

feature_tree_to_nl(snapshot)
  -> 输出自然语言摘要（供 LLM 快速理解）

get_editable_parameters(snapshot)
  -> 提取所有可编辑参数列表（供 LLM 结构化输出）
```

### editor.py — 局部修改指令生成 (230 行)

LLM 理解模型后定位到要修改的特征，只对该特征生成局部修改代码。
**不改的特征在 NX 中原样不变。**

**NX 参数修改的核心方式 — Expression 优先：**

```python
# 方式一（推荐）: 通过 Expression 修改 — 最通用、最可靠
expr = work_part.Expressions.FindObject("p5")
unit = work_part.UnitCollection.FindObject("MilliMeter")
work_part.Expressions.EditWithUnits(expr, unit, "25")
work_part.Update()
```

支持的操作：

| 操作 | 说明 | NX 实现方式 |
|------|------|------------|
| MODIFY_PARAM | 修改特征参数 | Expression EditWithUnits |
| REMOVE_FEATURE | 删除特征 | DeleteFeature |
| INSERT_AFTER | 在指定特征后插入 | Builder API + 重排序 |
| INSERT_BEFORE | 在指定特征前插入 | Builder API + 重排序 |
| SUPPRESS | 抑制特征 | feat.Suppressed = True |
| UNSUPPRESS | 取消抑制 | feat.Suppressed = False |
| RENAME | 重命名特征 | feat.Name = "new_name" |
| DUPLICATE_MODIFY | 复制并修改 | MirrorFeature / Pattern |

生成的代码示例：

```python
# 修改 EXTRUDE(1) 的拉伸深度
target_expr = work_part.Expressions.FindObject("p5")
unit_mm = work_part.UnitCollection.FindObject("MilliMeter")
work_part.Expressions.EditWithUnits(target_expr, unit_mm, "30")

# 抑制 HOLE(1)
feat = work_part.Features.FindObject("HOLE(1)")
if feat: feat.Suppressed = True

# 更新模型
work_part.Update()
```

### dependency.py — 依赖关系管理 (180 行)

管理特征之间的依赖关系图，逻辑与 CatiaCopilot 通用：

```
DependencyGraph
  build_from_features(features)   # 从特征列表构建依赖图
  topological_sort()              # Kahn 算法拓扑排序
  get_dependents(name)            # 获取直接依赖的特征
  get_all_dependents(name)        # BFS 递归获取所有间接依赖
  can_safely_remove(name)         # 判断删除是否安全
  get_insertion_points(new, deps) # 判断新特征的插入位置

EditValidator
  validate_edit(old, new)         # 验证编辑后的依赖完整性
```

### validator.py — NXOpen 代码验证 (100 行)

利用 Python ast 模块做语法检查（比 CATIA 的 VBS 验证更强大）：

```
NXCodeValidator
  validate_python_syntax(code)    # ast.parse 语法检查
  validate_nxopen_safety(code)    # 危险模式检测 (os.remove, subprocess, eval)
  validate_references(code, known)# FindObject 引用匹配检查
  validate(code, known_features)  # 综合验证
```

### forward_builder.py — 正向创建代码生成 (380 行)

从结构化 FeatureSpec 或自然语言描述，生成完整的 NXOpen Python 创建代码。

**12 个完整 NXOpen Builder 模板：**
extrude, revolve, hole, edge_blend, chamfer, shell,
circular_pattern, rectangular_pattern, mirror,
boolean_subtract, boolean_unite, datum_plane。

**自然语言解析支持：**
- "拉伸20mm" -> Extrude
- "打孔 直径10 深度25" -> Hole
- "倒圆角R5" -> EdgeBlend
- "倒角3mm" -> Chamfer
- "抽壳2mm" -> Shell
- "圆形阵列6个" -> CircularPattern
- "矩形阵列3x4" -> RectangularPattern
- 多特征组合: "拉伸20mm 然后打孔 直径10" -> [Extrude, Hole]

### 特征映射表 (feature_mapping.yaml)

定义了每种特征类型到 NXOpen API 的映射关系，覆盖 29 种常用特征类型：

```yaml
- feature_type: "Extrude"
  nxopen_class: "NXOpen.Features.Extrude"
  parameters:
    StartValue: { type: float, unit: "MilliMeter", default: 0 }
    EndValue: { type: float, unit: "MilliMeter", default: 20 }
  dependencies:
    - { type: "Sketch", required: true }
  creation_template: "extrude"

- feature_type: "Hole"
  nxopen_class: "NXOpen.Features.Hole"
  parameters:
    Diameter: { type: float, unit: "MilliMeter", default: 10 }
    Depth: { type: float, unit: "MilliMeter", default: 20 }
    TipAngle: { type: float, unit: "Degree", default: 118 }
  dependencies: []
  creation_template: "hole"
```

29 种特征分类：
- 实体: Extrude, Revolve, SweepAlongGuide, Loft
- 孔槽: Hole, Pocket, Slot, Groove
- 圆角倒角: EdgeBlend, Chamfer, FaceBlend, VariableRadiusBlend
- 壳筋: Shell, Rib, Stiffener, Boss
- 阵列: CircularPattern, RectangularPattern, Mirror
- 布尔: Unite, Subtract, Intersect
- 变换: Draft, Offset, Scale
- 基准: DatumPlane, DatumAxis, DatumCsys, Sketch

### 工作流 1：打开模型 -> 理解 -> 修改 (只改目标特征)

```

                                    7. LLM 定位 CircularPattern 和 Hole

                                       # 修改阵列数量
                                       expr = work_part.Expressions.FindObject("pattern_count")

                                       work_part.Expressions.EditWithUnits(expr, unit, "8")
                                       # 修改孔直径
                                       expr = work_part.Expressions.FindObject("hole_dia")
```

### 工作流 3：纯对话（NX 未连接）

```
用户                               NXCopilot
----                               ---------
1. "帮我写一个 NXOpen 脚本，创建一个 50x30x20 的长方体"
                                    2. LLM 生成完整 NXOpen Python 代码
                                    3. 返回代码，用户可在 NX Journal 中执行
4. "再加一个直径10的通孔"
                                    5. LLM 在原代码基础上追加 Hole 创建代码
                                    6. 返回完整脚本
```

## 六、关键技术决策

### Agent 进程位置

**同机运行**（NX 所在的 Windows 机器），两种模式：

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| 本地模式 | Agent 作为 NX Journal 脚本启动 (同进程) | 最佳性能，零延迟 |
| 远程模式 | Agent 独立进程，通过 NXOpen.RemoteSession 连接 | 独立部署，多 NX 实例 |

推荐本地模式：Agent 嵌入 NX 作为 Journal 脚本启动，
NXOpen.Session.GetSession() 获取全局 Session，所有 API 调用零开销。

### 执行策略：NXOpen 直接执行 vs 代码生成

| 场景 | 策略 | 说明 |
|------|------|------|
| 正向创建新特征 | ForwardBuilder 生成代码 -> exec() | 12 个 Builder 模板 |
| 逆向理解模型 | NXOpen 读取特征树 -> 代码分析 | 只读不改 |
| 局部修改已有特征 | Expression EditWithUnits | 不改的特征不动，只改目标 |
| 抑制/取消抑制 | feat.Suppressed = True/False | 一行代码 |
| 删除特征 | DeleteFeature | 需检查依赖 |
| 批量/离线 | 代码生成 -> 保存为 .py 文件 | 可在 NX Journal 中手动执行 |
| 容错 | exec 失败 -> 降级为代码输出 | 用户可手动执行并反馈 |

### 模型上下文同步 (最核心)

**原则：Agent 永远以 NX 的实时状态为准。**

每次自然语言交互前：

```
1. Agent 通过 NXOpen 读取当前 Part 完整特征树
2. 对比内部缓存的上一份快照 (如有)
3. DeltaDetector 计算增量差异 (新增/修改/删除)
4. 将差异注入到 LLM 上下文
5. LLM 基于真实状态进行推理
6. 执行完毕后更新快照
```

这样即使用户在 NX 中手工修改了模型，Agent 也能感知到变化。

**NX 的 Expression 优势：** NX 的 Expression 系统比 CATIA 的 Parameter 更统一，
所有参数化尺寸都可以通过 Expression 名称直接访问和修改，
不需要像 CATIA 那样通过复杂的 API 路径定位。

### LLM 使用策略

不自建 LLM 服务，通过第三方 API 调用：

| 提供商 | 适用场景 |
|--------|----------|
| DeepSeek (默认) | 性价比高，中文理解强，适合日常 CAD 辅助 |
| OpenAI (GPT-4o) | 需求理解、结构化指令输出 |
| Anthropic (Claude Sonnet) | 代码生成、复杂推理 |
| Ollama (本地) | 离线/本地调试备用 |

LLM 在流程中的角色：

```
用户需求 "把孔径改成12mm"
    |
    v
LLM API 调用 (第三方 SDK)
    -> 输出结构化指令:
       { action: "modify_param",
         feature: "HOLE(1)",
         param: "Diameter",
         value: 12.0 }
    |
    v
NXEditor / NXOpen 执行
    -> Expression EditWithUnits
    -> work_part.Update()
```

LLM 只做"理解需求 -> 输出结构化指令"，不做直接替换操作。

### NXOpen 代码验证优势

NXCopilot 使用 Python ast 模块做代码验证，比 CatiaCopilot 的 VBS 验证更强大：

```python
import ast
# 完整语法树解析
ast.parse(code)  # SyntaxError if invalid

# 危险模式检测
import re
re.search(r'os\.remove|subprocess', code)  # 危险操作拦截

# 引用匹配
re.findall(r'FindObject\("([^"]+)"\)', code)  # 提取所有特征引用
```

## 九、风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| NXOpen API 版本差异 (NX 12 vs NX 2306) | 部分 API 不兼容 | reader.py 做版本检测，关键调用 try/except 保护 |
| 特征参数路径不确定 | 参数读取失败 | Journal 录制 + 手动探索确认，逐步补充映射 |
| LLM 理解需求出错 | 改错特征 | LLM 只输出结构化指令，用户确认差异后执行 |
| 复杂特征创建代码不完整 | 正向设计受限 | 先覆盖 12 种高频特征模板，逐步扩展 |
| NX Session 断开 | Agent 无法操作 | executor.py 重试 + session.py 自动重连 |
| 完整重建导致特征改名/引用丢失 | 模型损坏 | **禁止完整重建**，全部使用局部修改 |

## 十、总结

NXCopilot 的三大能力，核心是 **NX 原生 Python 集成** 和 **Expression 参数化**：

```
                    正向设计
              +-------------------+
              | 自然语言 -> 建模   |
              | NXOpen Builder API |
              +---------+---------+
                        |
          +-------------+-------------+
          |  NX Expression 系统       |
          |  (参数化建模核心)          |
          +-------------+-------------+
                        |
          +-------------+-------------+
          |                           |
    +-----v-----+           +---------v---------+
    |逆向设计    |           | 混合设计           |
    |AI 读懂    |           | 手工 + AI 接续    |
    |当前模型   |           | 局部修改，不改的不动 |
    +-----------+           +-------------------+
```

**相比 CatiaCopilot 的核心优势：**

1. **零桥接开销** — NXOpen Python 是原生 API，不需要 COM/pywin32
2. **更强的代码验证** — Python ast 模块比 VBS 语法检查更可靠
3. **统一的参数系统** — Expression 覆盖所有参数化尺寸，修改路径最短
4. **更简的架构** — 无 MCP Server 层，Agent 直接调用 NXOpen

开发节奏：12 周完整，前 3 周完成框架 + Agent + UI (2,231 行代码，45 测试通过)。
阶段 4 (NX 环境实测) 是关键里程碑 — 在真实 NX 中验证完整链路。
